[1] Papel
Você é um engenheiro de software especializado em testes unitários e critérios de teste estrutural.

[2] Tarefa
Gerar testes unitários em Java (JUnit 5) para o método nextPrime(int n).
Não gerar testes para outros métodos da classe.

[3] Conformidade dos nomes dos casos de testes
Todos os casos de teste devem ter nomes que façam referência explícita ao método nextPrime,
como por exemplo: testNextPrime_*, nunca utilizando nomes de outros métodos ou termos não relacionados.

[4] Raciocínio Programático (Program of Thoughts)

Antes de gerar o código final, produza apenas um raciocínio estruturado em comentários iniciados com "#", seguindo as instruções abaixo. Não gere código Java nesta etapa.
 
4.1 — Aplicação dos Critérios de Fluxo de Dados

Aplique cuidadosamente os seguintes critérios, descrevendo seu raciocínio:

# Na geração de casos de teste, considere cuidadosamente as variáveis definidas e usadas no método nextPrime(int n),
# produzindo dados de teste que satisfaçam os seguintes critérios:

# All-Defs
# Gere casos de teste que exercitem todas as partes do programa onde uma variável recebe um valor (definição),
# garantindo que exista pelo menos um caminho de execução no qual essa definição alcance um uso válido da variável
# (em expressões, condições ou cálculos), sem que ocorra nova redefinição antes desse uso.

# All-Uses
# Gere casos de teste que percorram todos os caminhos de execução onde cada variável definida possa alcançar
# todos os seus usos possíveis:
# - usos computacionais (c-use), quando a variável é utilizada em cálculos ou expressões, e
# - usos predicativos (p-use), quando a variável é utilizada em condições de decisão ou estruturas de repetição.
#
# Cada par definição-uso deve ser exercitado pelo menos uma vez, garantindo que não ocorra nova redefinição da variável entre o ponto da definição e o ponto do uso.

# All DU-Paths
# Gere casos de teste que percorram todos os caminhos de definição para uso possíveis no programa, incluindo alterna# tivas de fluxo como: caminhos condicionais, caminhos que evitam redefinições intermediárias, caminhos que ocorrem # fora de laços ou que garantam pelo menos uma cobertura representativa dentro/fora de estruturas de repetição.

[5] Geração do Código Final (JUnit 5)
